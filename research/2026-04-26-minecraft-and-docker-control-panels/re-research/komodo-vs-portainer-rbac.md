### Re-research for: Komodo vs Portainer BE RBAC
Status: CONFIRMED-COUNTER

The original superlative ("most comprehensive of the open-source panels surveyed") does not survive a side-by-side reading of primary docs. Portainer BE is closed-source commercial — so it would be excluded from "open-source panels surveyed" — but on raw RBAC depth Portainer BE is at least Komodo's equal and beats it on several axes. The cited URL `https://komo.do/docs/resources/permissioning` is indeed dead (404 reproduced); the live page is `https://komo.do/docs/configuration/permissioning` [src:https://komo.do/docs/configuration/permissioning | authority:vendor primary docs].

**Per-resource granularity.** Komodo defines four hierarchical permission levels — None, Read, Execute, Write — applied per resource or per resource-type, with regex matching for bulk targeting (e.g., "give john execute on all of their Stacks"), plus specific feature gates for Logs, Inspect, Terminal, Attach, and Processes [src:https://komo.do/docs/configuration/permissioning | authority:vendor primary docs]. Portainer BE ships eight predefined roles — Administrator, Environment Administrator, Edge Administrator, Operator, Namespace Operator, Standard User, Helpdesk, Read-Only User — and adds resource-level access tiers (Administrator-only, Public, Restricted to specific teams/users) with parent-to-child inheritance through stacks/services and label-based control (`io.portainer.accesscontrol.public`, `.teams=`, `.users=`) for resources created outside Portainer [src:https://docs.portainer.io/admin/user/roles | authority:vendor primary docs] [src:https://docs.portainer.io/advanced/access-control | authority:vendor primary docs]. Portainer also explicitly supports Kubernetes namespace-scoped roles ("Namespace Operator … restricted to specified namespaces instead of the entire environment"), which Komodo's docs do not match.

**Group/team layering.** Komodo has User Groups with "Everyone" mode and Linux-style additive inheritance across multiple groups [src:https://komo.do/docs/configuration/permissioning | authority:vendor primary docs]. Portainer BE has Teams with role-per-environment assignment — "a single user or team can have different roles for different environments in the Portainer inventory" [src:https://docs.portainer.io/admin/user/roles | authority:vendor primary docs]. Both have group/team layering; Portainer's per-environment role binding is a dimension Komodo's docs don't mirror.

**OIDC / OAuth providers.** Komodo: GitHub, Google, generic OIDC, with documented setups for Authentik, Gitea, and Keycloak (env vars `KOMODO_OIDC_ENABLED`, `KOMODO_OIDC_PROVIDER`, etc.) [src:https://komo.do/docs/setup/advanced | authority:vendor primary docs]. Portainer: pre-configured Microsoft (Azure AD/Entra ID), Google, GitHub, plus a Custom provider for third-parties like Keycloak, alongside LDAP and Active Directory [src:https://docs.portainer.io/admin/settings/authentication/oauth | authority:vendor primary docs] [src:https://docs.portainer.io/admin/settings/authentication | authority:vendor primary docs]. Both cover the same OIDC ground; Portainer additionally bundles LDAP and AD, which Komodo's docs do not mention as native.

**Audit log.** Portainer BE has a first-class Activity log capturing date/time, user, endpoint, and action for every UI/API operation, with filtering, payload inspection, and CSV export — explicitly BE-only [src:https://docs.portainer.io/admin/logs/activity | authority:vendor primary docs]. Komodo's permissioning page does not document an equivalent audit-log surface.

**Bottom line.** "Most comprehensive" does not survive. The accurate framing is **comparable on per-resource granularity, with Portainer BE clearly ahead on (a) namespace-scoped roles, (b) per-environment role binding, (c) LDAP/AD on top of OIDC, and (d) a built-in audit log — and Komodo ahead on (a) regex-targeted permission grants and (b) the simpler Linux-style additive group model**. Also worth flagging: Portainer BE is commercial, not open source, so it's defensible to keep a softened claim scoped to *open-source* panels (e.g., "Komodo's RBAC is the deepest among the open-source panels surveyed; Portainer's equivalent depth requires the paid Business Edition") — but the unqualified superlative needs to go, and the dead URL must be replaced with `https://komo.do/docs/configuration/permissioning`.
