# Research: Minecraft + Docker control panels — landscape & build-on options
_Compiled 2026-04-26 via /research-here_

This document surveys Minecraft-specific panels, generic Docker panels, the overlap zone where the two collide, the architectural patterns that recur across both worlds, the licensing dynamics that shape forks, the practitioner pain points that show up repeatedly in issue trackers and forums, and the strategic options open to the `mcontrol` project. The user's `bserver` currently runs six Minecraft servers as per-server custom Docker Compose stacks, layered on either `eclipse-temurin:21-jre` or `itzg/minecraft-server:java21` (with itzg's entrypoint deliberately bypassed). The question this document informs is whether `mcontrol` should fork an existing panel, integrate against one as a backend, or remain a thin wrapper over the existing compose-and-socket pattern inherited from `admin_management`.

## Pterodactyl is the de-facto standard, but momentum is splintering across forks
**The MIT-licensed Panel + Wings split made Pterodactyl ubiquitous, and now that license is being exercised — Pelican and Pyrodactyl are both real, active forks competing on overlapping ground.**

Pterodactyl is MIT-licensed across both [Panel](https://github.com/pterodactyl/panel/blob/1.0-develop/LICENSE.md) and the Go [Wings daemon](https://github.com/pterodactyl/wings), shipping a PHP/Laravel + React/TS panel paired with a Docker-isolated agent. After a quiet stretch the project is shipping again as of 2026: v1.12.0 landed Jan 6, v1.12.1 Feb 14, and v1.12.2 on Mar 26 2026, with 22 commits and 18 merged PRs in the last 90 days against the [Panel repo](https://api.github.com/repos/pterodactyl/panel) — though Wings itself is much quieter, with a single commit since Jan 26 2026 and zero merged PRs in the same window. Practitioner sentiment captures the duality precisely: ["Pelican is the future approach probably but it's in alpha and a fork of Pterodactyl. Nothing compared to Pterodactyl's features, stability"](https://lowendspirit.com/discussion/9906/any-good-alternatives-to-pterodactyl) and ["Pterodactyl is basically what the industry uses"](https://lowendspirit.com/discussion/9906/any-good-alternatives-to-pterodactyl).

[Pelican](https://github.com/pelican-dev/panel) is the most consequential fork — it [relicensed MIT to AGPL-3.0 with two months of prior notice](https://pelican.dev/faq/), framed as ["Difference of visions, directions, and opinions"](https://pelican.dev/faq/) rather than a hostile split, and its named team (Lance Pioch, Michael Parker, Charles Morgan, Alex Vlad, boy132, Martin Oscar) is publicly listed on the [About page](https://pelican.dev/docs/about). Pelican adds [PostgreSQL/SQLite support, a Filament+Vite UI, OAuth, Cloudflare Turnstile, webhooks, RBAC, and a first-party plugin system](https://pelican.dev/docs/comparison/). It remains in beta as of 2026-04 (latest v1.0.0-beta33 on Feb 18 2026) with around 2k stars and ~6566 commits per the [GitHub repo](https://github.com/pelican-dev/panel), and the docs themselves [warn it is beta and "some things might change or break between beta versions"](https://pelican.dev/docs/panel/getting-started/). [Pyrodactyl](https://github.com/pyrohost/pyrodactyl) is a third active fork, positioned as smaller and faster than Pelican; it self-reports approximately 170× smaller initial download and 16× faster load times relative to Pelican — but the evidence is two imgur screenshots in the README with no third-party benchmark ([weak source](https://github.com/pyrohost/pyrodactyl)), with concrete safety/accessibility features named being SRI/integrity checks for production builds, full keyboard navigation, and screen-reader compatibility (no independent audit found).

## itzg/minecraft-server is the gravitational center of Docker-Minecraft, but its fat-entrypoint design fights opinionated panels
**The image's value lives in its env-var contract; any panel that overrides the entrypoint throws that value away.**

The [itzg/minecraft-server](https://github.com/itzg/docker-minecraft-server) image is opinionated and entrypoint-driven — it auto-installs versions, modloaders, and modpacks at startup, with [over 100 million Docker Hub pulls](https://hub.docker.com/r/itzg/minecraft-server). Its env-var model uses [`EULA=TRUE`, `/data`, and a `TYPE` selector for Vanilla/Forge/Fabric/Modrinth/CurseForge/FTB](https://docker-minecraft-server.readthedocs.io/en/latest/types-and-platforms/), with the [server.properties docs noting "all known server.properties entries can be managed by the environment variables"](https://docker-minecraft-server.readthedocs.io/en/latest/configuration/server-properties/). RCON is enabled by default to enable graceful shutdown and backup coordination, with a random per-startup default password on port 25575, and the [canonical console interface is `docker exec -i mc rcon-cli`](https://docker-minecraft-server.readthedocs.io/en/latest/sending-commands/commands/). The shipped [compose example](https://github.com/itzg/docker-minecraft-server/blob/master/docker-compose.yml) uses `stdin_open` + `tty` so `docker attach` works, mounting a named volume to `/data`. The de jure stability picture is weaker than the de facto adoption suggests — the [variables documentation page itself warns it "may be out-of-date or inaccurate"](https://docker-minecraft-server.readthedocs.io/en/latest/variables/), so there is no formal stability commitment behind the env-var contract.

This entrypoint design collides head-on with panels that want to inject their own startup command. PufferPanel's Docker mode [overrides image ENTRYPOINT with the panel-defined run command](https://docs.pufferpanel.com/en/3.x/environments/docker.html) — confirmed both in the 3.x docs and in [`servers/docker/docker.go`](https://github.com/pufferpanel/pufferpanel/blob/main/servers/docker/docker.go) — and by design expects thin base images (its own [runtime-images](https://github.com/pufferpanel/runtime-images) ship only `java`/`steamcmd`/`ubuntu`, and the [Minecraft template](https://github.com/pufferpanel/templates/blob/main/minecraft/minecraft.json) launches `java` directly). A fat self-launching image like itzg's would have all of its env-var-driven setup bypassed in that mode. The same dynamic is observable in the bserver setup itself: the user's custom `Dockerfile` ENTRYPOINT supersedes itzg's, demoting the image to "a glorified JRE base." Generic Docker panels like [Dockge](https://github.com/louislam/dockge) sidestep the conflict entirely — they expose a console via web terminal but have no Minecraft semantics, so running itzg under Dockge means authoring compose, mounting `/data`, and exec-ing `rcon-cli` from the terminal yourself.

## Generic Docker panels solve container ops, not Minecraft semantics
**Portainer, Komodo, and Dockge are excellent at the container layer; none of them know what a server.properties is or how to send RCON commands.**

[Portainer](https://github.com/portainer/portainer) is the heavyweight — it manages Docker, Swarm, Kubernetes, Podman, and ACI under [zlib license](https://github.com/portainer/portainer/blob/develop/LICENSE) for CE plus a Business Edition tier, with the [latest CE 2.39.1 LTS landing Mar 19 2026](https://github.com/portainer/portainer/releases/latest) and 37.3k stars. The [Server + Agent topology](https://docs.portainer.io/start/architecture) supports an Edge Agent variant that reverse-tunnels TLS outward. Portainer CE has only Administrator + Standard User roles, while [BE adds Operator, Helpdesk, Read-Only User, plus Environment Administrator, Edge Administrator, and Namespace Operator roles](https://docs.portainer.io/admin/user/roles). The [JSON REST API is gated by a per-user `X-API-Key` header](https://docs.portainer.io/api/access), with [worked examples](https://docs.portainer.io/api/examples) covering common operations.

[Komodo](https://github.com/moghtech/komodo) (formerly Monitor by mbecker20, [previously released as v1.13.0](https://github.com/mbecker20/monitor/releases/tag/v1.13.0)) is the most ambitious of the new wave — [GPL-3.0](https://github.com/moghtech/komodo/blob/main/LICENSE), Rust+TS Core+Periphery split, v2.1.2 on Apr 10 2026, 11k stars. The [primitive set](https://komo.do/docs/intro) covers container lifecycle, compose stacks with auto-deploy on git push, browser shell + container exec, log views, image builds (with optional AWS EC2 spot capacity), and Procedures/Actions for multi-step scheduled workflows plus an audit log. It treats infrastructure-as-code as first-class via TOML "Resource Sync" — `stacks.toml`, `repos.toml` — feeding a GitOps loop, with the canonical schema visible in the [periphery config](https://github.com/moghtech/komodo/blob/main/config/periphery.config.toml). Among open-source panels its [RBAC](https://komo.do/docs/configuration/permissioning) is the deepest (per-resource None/Read/Execute/Write, regex bulk targeting, additive User Groups), with OAuth (GitHub/Google) and generic OIDC (Authentik/Gitea/Keycloak documented) wired in via [advanced setup](https://komo.do/docs/setup/advanced) — though Portainer BE still exceeds it on namespace-scoped roles, per-environment role binding, LDAP/AD, and a built-in [Activity log](https://docs.portainer.io/admin/logs/activity).

[Dockge](https://github.com/louislam/dockge) is the deliberate small-tool counter-trend: MIT, by Louis Lam (also Uptime Kuma), v1.5.0 on Mar 30 2025, [MIT-licensed](https://github.com/louislam/dockge/blob/master/LICENSE). It is compose-only by design, with stacks living on disk (default `/opt/stacks`), an interactive web terminal, a `docker run` → `compose.yaml` converter, and explicitly no plugin or REST API surface. Multi-host arrived in [v1.4.0 on Jan 3 2025](https://github.com/louislam/dockge/releases) via "multiple Dockge instances." A real constraint to be aware of: [Docker Compose resolves relative volume paths against the stacks directory, and since the daemon runs on the host, host and container paths for the stacks dir must match](https://github.com/louislam/dockge/discussions/146). [Yacht](https://github.com/SelfhostedPro/Yacht) is dormant — MIT, [last release v0.0.7-alpha-hf-1 May 31 2021](https://github.com/SelfhostedPro/Yacht/tags), README warns "This applicaton has not been updated in a while." [Lazydocker](https://github.com/jesseduffield/lazydocker) is the TUI option — MIT Go, single-host, no RBAC, no web UI, v0.25.2 Apr 19 2026.

## Three architectures dominate: panel+daemon, single binary, and server+thin agent
**Architecture choice cascades into deployment complexity, language ecosystem, and what extension model is even possible.**

The classic [Pterodactyl shape](https://github.com/pterodactyl/panel) is Panel (PHP Laravel + React/TS) plus [Wings](https://github.com/pterodactyl/wings) (Go daemon, ~99.6% Go) where Wings exposes an HTTP API plus built-in SFTP for panel comms; extensibility lives at the JSON Egg catalog level rather than as code-level plugins. Pelican preserves the Panel + Wings + Node split and pins Wings versions to Pelican Panel. The same shape recurs in Komodo with different language choices: Core (REST + WebSocket API + UI) plus Periphery (a small stateless Rust agent on each host, bidirectional WebSocket from Core), roughly 60% Rust, 37% TS per the [intro docs](https://komo.do/docs/intro). PufferPanel went the other way and [collapsed panel + daemon into a single Go binary in v2.1.0](https://docs.pufferpanel.com/en/2.x/release-notes/2.1.0.html), config-toggled to act as panel, node, or both; [3.0 split panel/node Docker images and added template "conditions" for dual-environment templates](https://docs.pufferpanel.com/en/3.x/release-notes/3.0.0.html), with v3.0.7 released Apr 21 2026 from the [main repo](https://github.com/PufferPanel/PufferPanel).

[Crafty Controller 4](https://gitlab.com/crafty-controller/crafty-4) takes the opposite extreme — a single-process Python application on Tornado WebSocket server ([weak source - secondary](https://deepwiki.com/apdevstudio/crafty-controller)), Minecraft-only, GPLv3, [v4.10.4 on 2026-04-20](https://gitlab.com/api/v4/projects/crafty-controller%2Fcrafty-4/releases) under Arcadia Technology with ~6000+ commits and 68 releases since Aug 2020 per [the project page](https://craftycontrol.com/). [MCSManager](https://github.com/MCSManager/MCSManager) takes a different cut at the same problem — three-tier Frontend/Panel/Daemon over Socket.IO, no DB required, ~57% TS / ~40% Vue, Node ≥16.20.2 — and notably [pushes browser-to-daemon direct streams to bypass the panel for performance, which requires HTTPS reverse-proxying for each daemon](https://docs.mcsmanager.com/ops/mcsm_network.html). MCSManager is [Apache-2.0](https://github.com/MCSManager/MCSManager/blob/master/LICENSE) and the [README](https://github.com/MCSManager/MCSManager/blob/master/README.md) names an app marketplace, multi-node, multi-user, and a positioning around IDC commercial reselling. AMP (CubeCoders) sits outside this taxonomy because it is closed-source proprietary — only an [issue tracker on GitHub](https://github.com/CubeCoders/AMP) — and not a candidate to build on.

## License choice determines fork-friendliness — and Pelican proved relicensing is a viable path
**MIT lets a fork happen; AGPL §13 puts public-network deployments under a real obligation.**

Pterodactyl's MIT license is the only reason Pelican and Pyrodactyl exist as drop-in replacements rather than reimplementations. Pelican's [relicense to AGPL-3.0](https://pelican.dev/faq/) was deliberate and announced two months in advance; the practical effect is that [§13 imposes a mandatory obligation on any modified Pelican Panel exposed to remote users over a network — Corresponding Source must be offered](https://raw.githubusercontent.com/pelican-dev/panel/main/license). Pelican's optional commercial license is therefore only required to *buy* when all three of (panel modified) + (closed-source) + (publicly available) hold simultaneously. PufferPanel and MCSManager are Apache-2.0, friendliest of all for downstream productization. Crafty under GPLv3 and Komodo under GPL-3.0 require source disclosure for distribution but not for SaaS use. Portainer CE under [zlib](https://github.com/portainer/portainer/blob/develop/LICENSE) is permissive but its commercial-tier feature gating means RBAC depth past two roles is paywalled. AMP's pricing model — [Standard ~£7.50 (5 instances, standalone), Professional ~£15 (15 instances, Controller+Target), Advanced ~£30 (50 instances), Enterprise (variable), with only Enterprise permitting Commercial Usage](https://discourse.cubecoders.com/t/changes-to-amp-product-tiers-in-q3-2023/5178) and the [editions comparison sheet](https://discourse.cubecoders.com/t/editions-comparison-sheet/2247) — is one-time-payment lifetime-license but rules it out for any commercial reuse short of Enterprise.

## Real failure modes cluster around modpack memory, install friction, console stickiness, and abandoned panels
**Heavy-modpack workloads break panel assumptions; install steps break user patience; and panels go dark without warning.**

Pterodactyl modpack memory is a recurring open-issue theme: [#4447 reports `MaxRAMPercentage=95` causes modpacks like Divine Journey 2 to consume ~8GB instead of ~4GB and "Server will get killed by OOM killer as soon as a player interacts with it"](https://github.com/pterodactyl/panel/issues/4447). Wings itself has a [memory leak with high-file-count workloads (modpacks, dynmap), with #5063 reporting Wings climbing to 32GB on hosts with ~1M files — closed not-planned](https://github.com/pterodactyl/panel/issues/5063). Volume-permission issues recur — [#3040 documents `drwx------` and "bind source path does not exist" on Dockerized Wings](https://github.com/pterodactyl/panel/issues/3040). [Wings allocates both TCP and UDP for any allocated port even though Java MC only uses TCP — the UDP allocation is wasted](https://github.com/pterodactyl/panel/issues/4850). PufferPanel has its own console regression — [#1439 (Aug 2025): "Minecraft server console becomes unresponsive. Commands … stop appearing until the page is manually refreshed"](https://github.com/pufferpanel/pufferpanel/issues/1439). itzg has [an autopause-vs-watchdog interaction where the paused Java fails Minecraft's tick watchdog → "Considering it to be crashed, server will forcibly shutdown"](https://github.com/itzg/docker-minecraft-server/issues/3261).

Crafty's modpack story is essentially "do it manually" — [the server-builder UI only offers "Minecraft Java" or "Minecraft Bedrock"](https://docs.craftycontrol.com/pages/user-guide/server-creation/minecraft/), and per the [FAQ](https://docs.craftycontrol.com/pages/user-guide/faq/) Forge requires you to "set-up your forge server locally first, zip it up and import it into Crafty" while Modrinth `.mrpack` files "need[] to be unpacked... outside of Crafty and import[ed]." Pterodactyl modpack support is [community-egg territory: the official "CurseForge Generic" egg accepts a project ID, supports Forge/NeoForge/Fabric/Quilt, requires CurseForge API key](https://eggs.pterodactyl.io/egg/games-curseforge-generic/). AMP polarizes around install-vs-friction — praised for fast install (["up and running in 10–15 minutes"](https://forum.level1techs.com/t/amp-game-server-good-or-bad/231908)), Windows-native support, [responsive paid staff and one-time-payment lifetime license](https://discourse.cubecoders.com/t/why-did-you-pick-amp/11866); criticized for [multi-layer configuration model confusing newcomers](https://discourse.cubecoders.com/t/amp-is-not-for-me-gave-it-a-try-found-it-too-cumbersome-can-i-get-a-refund/4899), [schedule/auto-update triggers that "just don't work"](https://discourse.cubecoders.com/t/challenges-getting-started-with-amp/14132), role/permission UI complexity, Windows Base URI gotchas, and [ADS controller/target distributed-mode confusion](https://discourse.cubecoders.com/t/ads-deployment-modes/2245). Heavy self-hosters report migrating *from* Portainer back to Dockge: ["Portainer might be more than what I actually need. Yes, it's powerful, but it's simply got too many moving elements"](https://www.xda-developers.com/why-i-stopped-using-portainer-and-went-back-to-dockge/). And panels do go dark — Yacht's last release was 2021, and even Pelican install/migration friction is real enough that [community migration scripts exist](https://github.com/FinnAppel/Ptero-to-Pelican-Migration-Script).

## Five build-on paths exist for mcontrol; only two preserve scope-fit
**Given the bserver inventory, the honest recommendation is a thin Minecraft-aware UI over compose + itzg env vars — adopt itzg's contract instead of bypassing it.**

Path 1 — **Fork Pterodactyl or Pelican.** Gets a richer panel for free but inherits a maintenance burden in a [crowded fork landscape](https://github.com/pelican-dev/panel) where [Pterodactyl + Pelican + Pyrodactyl are all actively developed](https://github.com/pyrohost/pyrodactyl). Pelican has a real first-party plugin system (beta30+) and is explicit that ["Blueprint extensions are not compatible with Pelican"](https://pelican.dev/faq/), and the Blueprint creator declined to support Pelican; meanwhile [Blueprint](https://github.com/BlueprintFramework/framework) for Pterodactyl is active (1853 commits, release Jan 2026) but patches Pterodactyl files directly, making upgrades brittle. Forking buys features at the cost of becoming a fork-maintainer.

Path 2 — **Ship Pterodactyl/Pelican Eggs, no fork.** Use [the egg system as the extension target](https://pterodactyl.io/community/config/eggs/creating_a_custom_egg.html) — Configuration Files (parsers for YAML/JSON/properties/XML), Variables with validation, [`{{VARIABLE_NAME}}` substitution, "done" marker line](https://pterodactyl.io/community/config/eggs/creating_a_custom_egg.html), targeting [yolks images](https://github.com/pterodactyl/yolks) (e.g. [Paper egg uses `ghcr.io/ptero-eggs/yolks:java_25`](https://eggs.pterodactyl.io/egg/games-paper/)) on the [`container` user / `/home/container` WORKDIR convention](https://pterodactyl.io/community/config/eggs/creating_a_custom_image.html). Crucially, eggs are portable — [pelican-eggs preserves the JSON shape as PTDL_v2](https://github.com/pelican-eggs), Pelican exports eggs in Pterodactyl's own format ([sample paper egg](https://raw.githubusercontent.com/pelican-eggs/minecraft/main/java/paper/pterodactyl-egg-paper.json)), and the [Pelican FAQ confirms "the original ones will always work"](https://pelican.dev/faq/). This avoids the fork-maintainer trap but locks `mcontrol` into someone else's panel UI.

Path 3 — **Integrate against Portainer as a backend.** The [per-user RESTful API can act as a gateway to the underlying Docker/K8s API](https://docs.portainer.io/api/access), with [worked examples](https://docs.portainer.io/api/examples), so `mcontrol` could plausibly use Portainer as the orchestrator. But [API-created stacks have limited UI editability — long-standing issue #6481](https://github.com/portainer/portainer/issues/6481) — and this loses the Minecraft semantics that justify `mcontrol`'s existence in the first place.

Path 4 — **Build a Dockge plugin.** Impossible: Dockge has [no plugin or REST API surface](https://github.com/louislam/dockge), only "FS co-location" (write compose files into the stacks dir), and [stacks-dir-path collisions are a known pain](https://github.com/louislam/dockge/discussions/146). Likewise MCSManager exposes a [per-user API-key gated API](https://docs.mcsmanager.com/apis/get_apikey.html) but no documented plugin/extension architecture, and Crafty's [v2 REST API with bearer-token auth](https://wiki.craftycontrol.com/en/4/docs/API%20V2) ([weak source - URL redirects](https://wiki.craftycontrol.com/en/4/docs/API%20V2)) plus [Server Webhooks and Prometheus metrics](https://docs.craftycontrol.com/) is a usable surface but, again, locks UX into Crafty.

Path 5 (recommended) — **Thin Minecraft-aware UI over per-server compose + itzg's env-var contract, building on `admin_management`'s existing Docker socket pattern.** The bserver inventory shows the value being thrown away: itzg is mounted as a base image but its ENTRYPOINT is overridden, so all of EULA/TYPE/server.properties templating is bypassed. Reversing that — embracing itzg's contract for new servers, keeping the bind-mount + custom-compose pattern, exposing RCON via the existing `docker exec ... rcon-cli` convention — keeps `mcontrol` scope-fit (single host, six servers, focused MC semantics), keeps the Docker-socket access pattern already in production via `admin-dashboard`, and avoids both the fork-maintainer burden and the generic-panel semantic gap. Pterodactyl-style eggs can be borrowed as a *schema* for variables and config-file parsers without adopting the panel.

## Current bserver setup (informs the build decision)

- Six Minecraft servers live under `/home/abstract/servers/minecraft/<name>/`. Only `kobra_kollektivet` is currently healthy and running (Fabric 1.20.1, dynmap on 8123, voicechat on 24454/udp, `-Xmx12G`). `atm10` was created but never started; `monifactory` exited 137 (OOM); `skyblock_johanna` exited 143; `all_the_mons` exited.
- Per-server file pattern is consistent: `docker-compose.yml` (`build: .`, bind `./server` → `/data`, RCON enabled, container labeled `com.noelkleen.service=minecraft`), `Dockerfile` (`FROM <base>`; `COPY entrypoint.sh`; `ENTRYPOINT ["/entrypoint.sh"]`), `entrypoint.sh` (`cd /data && exec ./start_server.sh`), and `server/start_server.sh` containing the actual `java -Xmx12G -jar <loader>.jar nogui` invocation.
- Two base images are in use: `eclipse-temurin:21-jre` (kobra_kollektivet) and `itzg/minecraft-server:java21` (atm10, monifactory). The custom Dockerfile ENTRYPOINT *overrides* itzg's entrypoint, so itzg's auto-install / EULA / TYPE / server.properties templating is bypassed — itzg is being used as a glorified JRE base.
- `RCON_PASSWORD` is hard-coded as `"rconer"` in every compose file.
- All servers use bind mounts, not named volumes — host directories retain everything.
- No Docker memory limits are set; only JVM `-Xmx`.
- No JVM tuning beyond `-Xmx` (no Aikar's flags / G1GC tuning).
- Five stale Docker networks (`fitta_default`, `goated_default`, `pokimon_default`, `pungkula_default`, `testing_default`) survive from deleted servers.
- `docker compose ls` reports stale `/mc-servers/...` paths for `atm10`/`monifactory`/`skyblock_johanna` — those paths no longer exist on disk.
- The predecessor project `admin_management` runs as the `admin-dashboard` container on port 8000; it mounts `/var/run/docker.sock` and `/home/abstract/servers/minecraft → /mc-servers` into itself, backed by Postgres 17. `mcontrol` is the focused MC-only extraction.
- Portainer was previously installed: a `portainer_data` volume from 2025-04-28 exists, but no Portainer container is currently running.

## Red flags

- Hard-coded `RCON_PASSWORD="rconer"` across every server is a credential-rotation hazard if any port is ever exposed beyond loopback.
- Custom ENTRYPOINT pattern over `itzg/minecraft-server` discards the entire reason to use that image — every benefit it claims (auto-install, modpack TYPEs, server.properties templating, autopause coordination) is unreachable.
- No Docker `mem_limit` means a runaway modpack JVM can starve neighbours; `monifactory`'s 137 exit suggests this has already happened.
- Stale networks and stale `docker compose ls` registrations indicate no consistent teardown discipline; mirrors Portainer's [#6481 API-created-stack drift](https://github.com/portainer/portainer/issues/6481).
- Panel abandonment is real (Yacht 2021); choosing a single upstream to depend on carries real "panel goes dark" risk, especially for newer projects like Pelican still in beta.
- Wings-style memory leaks under high file count ([#5063](https://github.com/pterodactyl/panel/issues/5063)) and modpack `MaxRAMPercentage` foot-guns ([#4447](https://github.com/pterodactyl/panel/issues/4447)) mean even adopting Pterodactyl wholesale would not eliminate modpack-OOM pain.
- itzg's autopause feature [trips the Minecraft tick watchdog](https://github.com/itzg/docker-minecraft-server/issues/3261); useful to know before turning it on.
- itzg's variables documentation [self-disclaims accuracy and currency](https://docker-minecraft-server.readthedocs.io/en/latest/variables/), so any abstraction layer over those variables needs its own pin/version-test discipline.
- Pelican's plugin ecosystem maturity is unverified — choosing it for plugin extensibility may be premature.

## Open questions

- How large and high-quality is the Pelican plugin ecosystem in practice, compared to Blueprint's Pterodactyl ecosystem? No verified catalog count exists yet.
- Does Pyrodactyl ship its own plugin system, or does it inherit Pterodactyl's egg model unchanged?
- Are Pyrodactyl's claimed "170× smaller / 16× faster" numbers reproducible? They have never been independently benchmarked.
- What is Crafty's outside-PR acceptance velocity? Currently unverified.
- What is the *de facto* stability of itzg's env-var contract over time, given the docs page itself disclaims accuracy? No formal commitment exists.
- Does MCSManager have a Chinese-language plugin ecosystem that is not surfaced in English-language docs?
- Of the cited GitHub issues (Pterodactyl #4447, #5063, #3040, #4850; PufferPanel #1439; Portainer #6481; itzg #3261), how many are still open vs resolved as of 2026-04? Citing closed issues as current pain points would be misleading.
