# mcontrol — Domain Glossary

Canonical vocabulary for `mcontrol`. Use these terms exactly in issues, PRs, commit messages, and code comments. When a concept isn't here, it may belong in a new decision entry rather than in a synonym invented on the fly.

For architectural decisions cross-referenced below, see [`decisions.md`](decisions.md).

---

## server

A single Minecraft server instance managed by mcontrol. Each server is represented by:

- A directory on the host at `<SERVER_BASE_PATH>/<name>/` containing the compose file and a `server/` data subdirectory (decision 008).
- A row in `app_mcontrol.servers` keyed by `name`.
- A Docker container whose name resolves via `db.container_name_for(row)` — either the `container_name` override or the row's `name` (decision 021).

"Server" always refers to this compound entity (directory + DB row + container), not to any one component alone.

## server base path

The host directory that contains all per-server directories. Configured via the `SERVER_BASE_PATH` environment variable. Typically `/home/abstract/servers/minecraft/` on bserver. Discovery walks this path at app startup (and on `POST /rescan`) to sync the DB with the filesystem (decision 034).

## bind mount

Each server's `<dir>/server/` subdirectory is bind-mounted into the container at `/data`. World data, jars, mods, configs, and `start_server.sh` all live inside this bind mount, making them accessible from the host without touching the container image (decisions 008, 023).

## world

The save data for a Minecraft server — chunk files, player data, level metadata. Lives inside `<dir>/server/<world-name>/` (e.g. `server/world/`, `server/world_nether/`). mcontrol never touches world data directly; it is strictly operator territory.

## jar

A Java archive file. In mcontrol's context, the **server jar** is the Minecraft server executable (e.g. `paper-1.21.4.jar`, a Forge/NeoForge loader jar, a Fabric launcher jar) that `start_server.sh` invokes. The filename is stored in `servers.variables.server_jar`. Operators upload jars via the file browser UI; mcontrol provides no auto-installer (decision 012).

## loader

A Minecraft modding framework — Forge, NeoForge, Fabric, Quilt, Paper, etc. — that is installed into the server jar. mcontrol does not automate loader installation; the operator supplies the loader jar directly (decision 012). The loader version is implicit in the jar filename; mcontrol does not parse or validate it.

## modpack

A curated collection of mods distributed as a unit (e.g. All the Mods 10 / `atm10`, Monifactory). mcontrol treats modpacks the same as any other server: operator uploads the loader jar and mods directory; mcontrol starts the container. No `.mrpack` parsing or Modrinth/CurseForge integration (decision 012).

## scaffold / scaffolded server

A server whose directory and compose file were generated by mcontrol's scaffolding flow (decision 023). Scaffolded servers have no per-server Dockerfile — the compose file references `eclipse-temurin:21-jre` directly and runs `./start_server.sh` from the bind-mounted `server/` directory. `app_mcontrol.servers.scaffolded_at` is non-null for scaffolded rows. Legacy servers (created before slice 6) have `scaffolded_at IS NULL` until the operator runs the one-shot migration (decision 028).

## start_server.sh

A shell script at `<dir>/server/start_server.sh` (inside the bind mount) that invokes the Java process: `java -Xmx<N>g [extra_args] -jar <server_jar> nogui`. Generated by mcontrol from a template and regenerated via the Variables card's Regenerate flow (decision 025). Operators may hand-edit it; a Regenerate clobbers hand-edits after a diff-preview confirmation.

## variables

Per-server runtime knobs stored in `app_mcontrol.servers.variables` as a JSONB column (decision 013). The canonical set:

| Key | Meaning |
|---|---|
| `memory_budget_gb` | Total memory allocation; drives both `-Xmx` and `mem_limit` (decision 009) |
| `motd` | Minecraft server message-of-the-day |
| `port` | Host-side TCP port for the Minecraft protocol (default 25565) |
| `server_jar` | Filename of the server jar in `server/` |
| `jvm_extra_args` | Additional JVM flags appended after `-Xmx` |
| `rcon_enabled` | Bool; effectively always true for mcontrol-managed servers |

## memory budget

A single integer (gigabytes) that mcontrol uses to derive both the JVM heap cap (`-Xmx`) and the container memory limit (`mem_limit`). The JVM heap is set to `budget − 2 GB` to leave headroom for off-heap memory (JIT code cache, native libs, Netty buffers). Stored as `variables.memory_budget_gb` (decision 009).

## RCON

Remote CONsole — a Source-engine-derived TCP protocol that Minecraft servers expose when `enable-rcon=true` in `server.properties`. mcontrol uses RCON to send commands to a running server (whitelist changes, op changes, console commands). The RCON password lives only in `server.properties`, managed by the operator (decision 024). If RCON is disabled, lifecycle / logs / files continue to work; only the RCON console tab is affected.

## lifecycle state

The current operational state of a server's Docker container, as returned by the Docker API and stored in `app_mcontrol.servers.state`. The states mcontrol recognises:

| State | Meaning |
|---|---|
| `created` | Container exists but has never been started |
| `running` | Container process is live |
| `paused` | Container frozen (rare; treated like `running` for button state) |
| `restarting` | Docker is cycling the container |
| `exited` | Container stopped (clean or crash) |
| `dead` | Docker marked the container dead; usually requires a force-remove |
| `scaffolding` | mcontrol is in the process of creating the server (transient) |
| `unknown` | Docker daemon was unreachable at last discovery scan |

`lifecycle_state.py` maps these states to the Start / Stop / Restart button enable/disable and accent (decision 033).

## discovery

The process of walking `SERVER_BASE_PATH`, querying Docker for container states, and syncing the result into `app_mcontrol.servers`. Runs once at app startup via FastAPI's lifespan handler, and on demand via `POST /rescan` (decision 034). Discovery inserts new rows for unknown directories and updates `state` for existing rows; it never overwrites operator-editable columns (`dir`, `container_name`) after initial insert (decision 021).

## tombstone

A soft-deleted server directory. When an operator deletes a server through the panel, mcontrol renames `<base>/<name>/` to `<base>/.deleted-<name>-<unix-ts>/` and removes the DB row. The data is not destroyed — the operator can recover it by renaming the directory back from a host shell. Discovery ignores dot-prefixed directories, so tombstones are invisible to scans. The `/trash` page lists tombstones and provides bulk-purge (≥ 7 days old) and per-tombstone delete affordances (decisions 026, 031).

## roster

The operator-trusted list of Minecraft player identities, stored in `app_mcontrol.players(uuid, name, added_at)`. Adding a player to the roster performs a Mojang API lookup to verify the UUID. Per-server membership (whitelist, ops) lives only on disk in each server's `whitelist.json` and `ops.json` — the roster is cross-server identity; membership is per-server and disk-authoritative (decision 027).

## whitelist / ops

`whitelist.json` and `ops.json` are the on-disk Minecraft files that control which players may join (`whitelist.json`) and which have elevated permissions (`ops.json`). mcontrol manages them via RCON when the server is running, or by direct atomic JSON read-modify-write when the server is offline. The panel never exposes a level dropdown for ops — vanilla `/op` always grants level 4, and `ops.json` cannot hot-reload non-default levels without a restart (decision 027).

## slice

A named, shippable vertical increment of the v1 build. Each slice has its own plan document in `docs/plans/` and produces working, testable software on its own. Slices are numbered monotonically (slice 1, slice 2, …) and land in order. References like "slice 6 PR 0" denote the first PR within a multi-PR slice.

## tailnet

The Tailscale private network that gates access to mcontrol. The panel is reachable only from devices on the operator's tailnet; there is no public ingress (decision 003). "On-tailnet" and "off-tailnet" are the access states; there is no app-level login.

## terminator

The TLS-terminating reverse proxy that sits in front of the panel. In production this is aserver-nginx, which holds the Let's Encrypt cert and proxies over the tailnet to bserver's LAN-bound app port (decision 019). mcontrol itself serves plain HTTP; it relies on the terminator for HTTPS.

## regenerate

The Variables-card affordance that re-renders `docker-compose.yml` and `start_server.sh` from templates and clobbers the on-disk files after a diff-preview confirmation. The diff endpoint captures file mtimes; the confirm endpoint re-stats them before writing and aborts if either file changed since the diff was shown (decision 025). Regenerate is the correct term — not "re-scaffold," not "re-generate," not "rebuild."
